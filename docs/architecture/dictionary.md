## Термины

### Шаблон термина
**Описание:** Тут должно быть полное описание термина.  
**Атрибуты:** Тут должны быть перечислены все атрибуты термина, с кратким описанием каждого из них. Например:
- `attribute1`: описание атрибута 1 
- `name`: имя человека  
- `age`: возраст человека

**Связи:** `relatedTerm1`, `relatedTerm2` - тут должны быть перечислены все связанные термины.  
**Жизненный цикл:** например: Created → Modified → Deleted - тут должен быть описан жизненный цикл термина, если он есть.   
**Инварианты:** это условия, которые всегда должны быть соблюдены для данного термина. Например:    
- Условие 1 
- age должен быть больше 0  
- name не должен быть пустым

**Зачем нужен:** тут должно быть описано, зачем нужен данный термин в контексте проекта.    
**Хранение:** тут должно быть описано, где и как будет храниться информация, связанная с данным термином. Например: в базе данных, в виде JSON-файла и т.д.     
**Формат:** тут должен быть описан формат данных, связанный с данным термином. Например: строка, число, дата, файл в формате pdf и т.д.     
**Источники:** тут должны быть перечислены источники, откуда будет поступать информация, связанная с данным термином. Например: пользовательский ввод, внешние API, файлы и т.д.    
**Примечание** - тут можно добавить любые дополнительные сведения, которые могут быть полезны для понимания термина. Например: примеры использования, ссылки на документацию и т.д. 
**См:** тут можно добавить ссылки на другие файлы документации, которые могут быть полезны для понимания термина. Например: [data-description.md](./data-description.md), и т.д.

### Данные планирования
**Описание:** Данные планирования, включающие название мультипроекта и горизонт планирования.  

**Атрибуты:**   
- `Название мультипроекта`: строка (например: "Цифровая трансформация")
- `Горизонт планирования`: дата окончания планового периода в формате дд.мм.гггг (например: "01.01.2024")

**Связи:** `Реализуемость портфеля по срокам`, `Портфельные ограничения`  

**Инварианты:**  
- `Название мультипроекта` не должно быть пустым
- `Горизонт планирования` не должен быть пустым, дата должна быть в формате дд.мм.гггг (например, 01.01.2024)

**Зачем нужен:** Для идентификации мультипроекта в пользовательском интерфейсе и определения даты окончания горизонта планирования, с которой будут сравниваться даты портфелей.

**Хранение:** В базе данных.   

**Источники:** Пользовательский ввод на фронтенде через форму.

### Стратегические ориентиры
**Описание:** Стратегические ориентиры компании описывают к чему стремится компания и какой приоритет имеет каждый ориентир.

**Атрибуты:** 
- `Список ориентиров`: Динамический список стратегических ориентиров

**Связи:** `Ориентир`, `Цель`.

**Инварианты:** 
- Не должен быть пустым, содержать хотя бы 1 ориентир

**Зачем нужен:** Для хранения информации о стратегических ориентирах компании.  

**Хранение:** в базе данных.

**Формат:** JSON-структура, которая содержит список стратегических ориентиров.     

**Источники:** Пользовательский ввод на фронтенде через форму.

### Ориентир
**Описание:** Ориентир компании, отвечает на вопрос "к чему стремимся?" и хранит приоритет ориентира.  

**Атрибуты:**
- `Видение`: Видение компании, отвечает на вопрос "к чему стремимся?", например: Сделать цифровую трансформацию доступной для среднего бизнеса к 2027 году.
- `Приоритет`: Приоритет ориентира (высокий, средний, низкий)

**Связи:** `Стратегические ориентиры`

**Инварианты:** 
- `Видение` не должно быть пустым
- `Приоритет` не должен быть пустым, должен быть один из трех значений: высокий, средний, низкий

**Зачем нужен:** для хранения информации о конкретном стратегическом ориентире.

**Хранение:** в списке стратегических ориентиров.

**Формат:** JSON-структура, которая содержит видение компании и приоритет ориентира.     

**Источники:** Пользовательский ввод на фронтенде через форму.

### Реестр проектов
**Описание:** Это перечень проектов в мультипроекте.  

**Атрибуты:**
- `Список проектов`: динамический список, который может содержать 1 или более проектов.

**Связи:** `Проект`, `Зависимости проектов`, `Зависимость`, `Граф зависимостей проектов`, `Цель`, `Альтернативы`, `План`, `Покрытие проектами`.  

**Инварианты:**    
- `Список проектов` Не должен быть пустым, должен содержать хотя бы один проект.

**Зачем нужен:** Для хранения информации о проектах, которые входят в мультипроект.    

**Хранение:** В базе данных. 

**Формат:** JSON-структура, которая содержит список проектов.     

**Источники:** Пользовательский ввод на фронтенде через форму.

### Проект
**Описание:** Это проект в мультипроекте.

**Атрибуты:** 
- `Название`: название проекта 
- `Статус`: статус проекта (Завершено, В работе, Не начато) 
- `Дата начала`: дата начала проекта в фомате дд.мм.гггг, наприме (01.01.2023)
- `Дата окончания`: дата окончания проекта, в формате дд.мм.гггг, например (01.01.2023)
- `Трудоемкость: аналитики`: запланированные трудозатраты в чел.часах по аналитике
- `Трудоемкость: разработчики`: запланированные трудозатраты в чел.часах по разработке
- `Трудоемкость: тестировщики`: запланированные трудозатраты в чел.часах по тестированию
- `Ограничения проекта`: текстовое описание ограничений проекта 
- `Текущие отклонения`: текстовое описание текущих отклонений
- `Описание проекта`: текстовое описание проекта 

**Связи:** `Реестр проектов`.

**Инварианты:**   
- `Название` не должно быть пустым 
- `Статус` не должен быть пустым, должен быть один из трех значений: Завершено, В работе, Не начато.
- `Дата начала` не должно быть пустым, должен быть записан в формате дд.мм.гггг (например, 01.01.2023)
- `Дата окончания` не должно быть пустым, должен быть записан в формате дд.мм.гггг, (например, 01.01.2023)
- `Трудоемкость: аналитики` не должно быть пустым, должно быть больше 0, целое число
- `Трудоемкость: разработчики` не должно быть пустым, должно быть больше 0, целое число
- `Трудоемкость: тестировщики` не должно быть пустым, должно быть больше 0, целое число

**Зачем нужен:** Для хранения информации о конкретном проекте.    

**Хранение:** в списке `Реестр проектов`   

**Формат:** JSON-структура, которая содержит информацию о проекте.     

**Источники:** Пользовательский ввод на фронтенде через форму.

### Зависимости проектов
**Описание:** Список зависимостей между проектами, показывает связь между проектами и ее направление.   

**Атрибуты:**   
- `Зависимость`: Зависимость между двумя проектами.

**Связи:** `Реестр проектов`, `Зависимость`, `Граф зависимостей проектов`, `Выявленные риски`, `Реализуемость портфеля по срокам`, `Анализ текущих отклонений`, `ИИ-объяснения`, `ИИ-рекомендации`, `Сценарий`.  

**Зачем нужен:** Хранит и отображает список зависимостей между проектами, используется в `Граф зависимостей проектов`, `Выявленные риски`, `Реализуемость портфеля по срокам`, `Анализ текущих отклонений`, `ИИ-объяснения`, `ИИ-рекомендации`, `Сценарий`. 

**Хранение:** в базе данных.

**Формат:** JSON-структура, которая содержит список зависимостей между проектами.    

**Источники:** Пользовательский ввод на фронтенде через форму.

### Зависимость
**Описание:** Зависимость между проектами, показывает связь между проектами и роль проектов в ней (основной, зависимый).   

**Атрибуты:**   
- `Основной проект`: Основной проект
- `Зависимый проект`: Проект, зависящий от основного
- `Описание связи`: Текстовой описание направляения зависимости, например Проект А зависит от Проекта B

**Связи:** `Зависимости проектов`.  

**Инварианты:**    
- `Основной проект` не должен быть пустым, не может быть равен `Зависимому проекту`, не может зависеть от `Зависимого проекта`
- `Зависимый проект` не должен быть пустым
- `Описание связи` не должно быть пустым

**Зачем нужен:** Для хранения информациb о зависимости между проектами.    

**Хранение:** в списке `Зависимости проектов`.     

**Формат:** JSON-структура, которая содержит информацию о зависимости между проектами.   

**Источники:** Пользовательский ввод на фронтенде через форму.

**Примечание:** `Описание связи` формируется автоматически с помощью ИИ на основе указаннх проектов (Основной, Зависимый).

### Портфельные ограничения
**Описание:** Хранит ограничения о доступных ресурсных лимитах (аналитиков, разработчиков, тестировщиков) и критическом сроке завершения мультипроекта. Используются при расчете `Реализуемость портфеля по срокам`, `Загрузка и дефицит ресурсов по ролям`, `Анализ портфельных ограничений`, `Сценарий`, `ИИ-рекомендации`.

**Атрибуты:**   
- `Лимит: аналитики`: лимит аналитиков (чел.час)
- `Лимит: разработчики`: лимит разработчиков (чел.час)
- `Лимит: тестировщики`: лимит тестировщиков (чел.час)
- `Критический срок`: критический срок завершения мультипроекта, в формате дд.мм.гггг

**Связи:** `Реализуемость портфеля по срокам`, `Загрузка и дефицит ресурсов по ролям`, `Анализ портфельных ограничений`, `Сценарий`, `ИИ-рекомендации`.

**Инварианты:**   
- `Лимит: аналитики`: не должен быть меньше 0
- `Лимит: разработчики`: не должен быть меньше 0
- `Лимит: тестировщики`: не должен быть меньше 0
- `Критический срок` должен быть в формате дд.мм.гггг

**Зачем нужен:** Отображает ограничения о доступных ресурсах и критическом сроке завершения мультипроекта. Данные будут использоваться при составлении альтернатив, формулировки рисков и предупреждений, ИИ-рекомендаций.  

**Хранение:** в базе данных. 

**Формат:** JSON-структура, которая содержит ограничения о доступных ресурсах и критическом сроке завершения мультипроекта.     

**Источники:** Пользовательский ввод на фронтенде через форму.

**Хранение:** в базе данных.

### Загрузка и дефицит ресурсов по ролям 
**Описание:** Показывает разницу между суммарной потребностью проектов, входящих в мультипроект и количеством доступных ресурсов.

**Атрибуты:** 
- `Роль`: Название роли
- `Суммарная потребность`: Суммарная потребность ресурсов
- `Лимит`: Лимит ресурсов
- `Дефицит/Профицит`: Дефицит/Профицит ресурсов

**Связи:** `Реестр проектов`, `Портфельные ограничения`, `План`.

**Инварианты:**
- `Роль` не должна быть пустой.
- `Суммарная потребность` не должна быть пустой, должна быть положительная и неотрицательная.
- `Лимит` не должна быть пустой, должна быть положительная и неотрицательная.
- `Дефицит/Профицит` не должна быть пустой, целое число.

**Зачем нужен:** Для отображения дефицита или профицита ресурсов на фронтенде.

**Хранение:** в базе данных.

**Формат:** JSON-структура, которая содержит информацию о ресурсах, запасах и загрузке. 

**Источники:** расчет на бэкенде.

**Источники:** ввод пользователя на фронтенде, формирование описания с помощью обращения к ИИ.

### Отклонение
**Описание:** Описание отклонения по проекту.

**Атрибуты:**
- `Проект`: Проект

- `Описание`: Текстовое описание сценария
- `Статус`: результат допустимости сценария, например Допустимый, Рекомендуемый, Требует внимания.
- `Интегральная оценка`: итоговая оценка сценария по совокупности критериев.
- `Общий срок`: общий срок выполнения портфеля
- `Количество рисков`: количество рисков, связанных со сценарием.
- `Проекты`: Список проектов в портфеле
- `Соблюдение ограничений`: процент или оценка соблюдения ограничений.
- `Ресурсная реализуемость`: степень соответствия потребности в ресурсах доступным лимитам.

- `Сильные стороны`: список преимуществ сценария.
- `Слабые стороны`: список недостатков сценария.
- `Перечень рисков`: риски сценария.
- `Соблюденные ограничения`: ограничения, которые выполняются в сценарии.

- `Количество рисков`: количество рисков, связанных со сценарием.
- `Соблюдение ограничений`: процент или оценка соблюдения ограничений.
- `Ресурсная реализуемость`: степень соответствия потребности в ресурсах доступным лимитам.
- `Общий срок реализации сценария`: общий срок выполнения портфеля.

### Паспорт стратегического плана 
**Описание:** сводная информация о плане, включающая его цель, приоритеты, объемы и сроки.

**Атрибуты:** 
- `Выбранный вариант`: выбранный вариант сценария (сбалансированный, консервативный, рискованный).
- `Плановый горизонт`: плановый горизонт в месяцах (т.е. на сколько месяцев вперед построен план).
- `Контрольных точек`: количество контрольных точек.
- `Рисков`: Количество рисков.
- `Ресурсы по ролям`: количество необходимых ресурсов по ролям (аналитики, разработчики, тестировщики).
- `Ограничения в зоне внимания`: количество ограничений в зоне внимания.
- `Прогресс выполнения`: Прогресс выполнения этапов плана.

**Связи:** `План`, `Сценарий`, `Риски плана`, `Ограничения в зоне внимания`.

**Инварианты:** 
- `Выбранный вариант` должен быть из списка `Альтернативы`.
- `Плановый горизонт` не может быть меньше 0.
- `Контрольных точек` не может быть меньше 0.
- `Рисков`: не может быть меньше 0.
- `Ресурсы по ролям`: не может быть меньше 0.
- `Ограничения в зоне внимания`: не может быть меньше 0.
- `Прогресс выполнения`: не может быть меньше 0 и больше 100, измеряется в процентах.

**Зачем нужен:** Для быстрого просмотра состояния стратегического плана без изучения всех этапов.

**Хранение:** подтягивается из данных плана, данные плана хранятся в БД.

**Источники:** данные из плана, расчет на бэкенде и ИИ-интерпретация.

### План 
**Описание:** итоговый стратегический план реализации мультипроекта, сформированный на основе выбранного сценария. Он включает в себя этапы реализации, контрольные точки, ресурсы, риски и ограничения в зоне внимания.

**Атрибуты:** 
- `Этапы реализации плана`: список этапов.
- `Контрольные точки`: список контрольных точек.
- `Риски плана`: перечень рисков.
- `Ограничения в зоне внимания`: ограничения, которые требуют контроля.
- `Выбранный сценарий`: сценарий, на основе которого построен план.
- `Загрузка ресурсов`: таблица загрузки ресурсов по этапам и по ролям.

**Связи:** `Сценарий`, `Паспорт стратегического плана`, `Этап`, `Риски плана`, `Этапы реализации плана`.

**Инварианты:** 
- `Этапы реализации плана`: не менее 1 этапа
- `Контрольные точки`: не менее 1 контрольной точки
- `Ограничения в зоне внимания`: не менее 1 ограничения
- `Выбранный сценарий`: в соответствии со сценариями в списке `Альтернативы`.
- `Загрузка ресурсов`: от 0%, может быть больше 100%.

**Зачем нужен:** Для фиксации итогового варианта реализации мультипроекта в виде управляемого плана.

**Хранение:** в базе данных.

**Формат:** JSON-структура стратегического плана.

**Источники:** выбранный сценарий и расчет на бэкенде, формирование описания с помощью обращения к ИИ.   

---
### Этапы реализации плана
**Описание:** список этапов реализации мультипроекта.

**Связи:** `Этап`, `План`.

**Атрибуты:** 
- `Этапы реализации плана`: список этапов.

**Инварианты:** 
- `Этапы реализации плана`: не менее 1 этапа, не должны противоречить зависимостям между проектами.

**Зачем нужен:** Для поэтапного представления реализации мультипроекта.

**Хранение:** в структуре `План`.

**Формат:** JSON-структура списка этапов.

**Источники:** выбранный сценарий и расчет на бэкенде, формирование описания с помощью обращения к ИИ.

---
### Этап
**Описание:** часть стратегического плана, ограниченная сроками и завершающаяся контрольной точкой. Этап содержит набор работ по проектам и необходимые ресурсы.

**Связи:** `Этапы реализации плана`, `План`, `Состав этапа`, `Контрольная точка`.

**Атрибуты:**
- `Номер этапа`: порядковый номер этапа
- `Название`: Название этапа, например "Запуск ключевых инициатив"
- `Период`: срок выполнения этапа, например Июль 2026 — Август 2026
- `Статус этапа`: Состояние этапа (Выполнен, В работе, Планируется)
- `Контрольная точка`: Результат выполнения этапа
- `Доля ресурса`: Доля ресурса, необходимая для выполнения этапа (чел.час)
- `Проекты этапа`: Список проектов, которые входят в этап

**Инварианты:** 
- `Номер этапа` Порядковый номер этапа
- `Название` Не должен быть пустым
- `Период` Значение должно быть в формате "месяц год - месяц год", например (Май 2026 - Июль 2026), начальная дата меньше конечной
- `Статус этапа` Статус этапа (Выполнен, В работе, Планируется)
- `Контрольная точка` Не должна быть пустой
- `Проекты этапа` Не должно быть пустым

**Зачем нужен:** Для структурирования плана на управляемые временные части.

**Хранение:** в структуре `Этапы реализации плана`.

**Формат:** JSON-структура этапа.

**Источники:** выбранный сценарий и расчет на бэкенде, формирование описания с помощью обращения к ИИ.

---
### Состав этапа 
**Описание:** перечень проектных работ, выполняемых в рамках конкретного этапа.

**Атрибуты:** 
- `Проект`: Проект, который входит в этап
- `Доля проекта`: Доля проекта (процент выполнения), входящего в этап 
- `Ресурс`: Ресурс (роль), необходимый для выполнения доли проекта 
- `Доля ресурса`: Доля ресурса, необходимая для выполнения доли проекта в составе этапа (чел.час)

**Связи:** `Этап`, `План`.

**Инварианты:** 
- `Проект` Не должен быть пустой, должен быть из `Реестр проектов`
- `Доля проекта` Доля проекта не должна быть меньше 0 и не больше 100

Зачем нужен:** Для детализации того, какие проектные работы выполняются на конкретном этапе.

**Хранение:** внутри объекта `Этап`.

**Формат:** JSON-список проектов этапа    

**Источники:** ИИ на основе данных о выбранном сценарии, проектах, ресурсах и ограничениях.   

---
### Контрольная точка
**Описание:** список ключевых результатов, по которым контролируется выполнение этапов стратегического плана.

**Атрибуты:** 
- `Название`: Название контрольной точки
- `Дата`: Дата контрольной точки
- `Статус`: Состояние контрольной точки (Достигнута, Запланирована)

**Связи:** `Этап`, `План`.

**Инварианты:** 
- `Название` Не должен быть пустым
- `Дата` Значение должно быть в формате дд.мм.гггг, например (01.01.2026)
- `Статус` Статус контрольной точки (Достигнута, Запланирована)

**Зачем нужен:** Для контроля выполнения плана и фиксации значимых промежуточных результатов.

**Хранение:** внутри объекта `План`.

**Формат:** JSON-список контрольных точек.

**Источники:** выбранный сценарий и генерация ИИ.

---
### Состояние этапов планирования 
**Описание:** статус прохождения основных этапов работы системы: загрузка контекста, анализ, определение целей, выбор альтернативы и формирование плана.

**Атрибуты:** 
- `Название`: Название функционального этапа
- `Статус`: Состояние этапа (Завершено, В процессе, Не начато)

**Инварианты:** 
- `Название` Не должен быть пустым
- `Статус` Статус этапа (Завершено, В процессе, Не начато)

**Зачем нужен:** Для контроля выполнения основных этапов работы системы.

**Хранение:** в базе данных.

**Формат:** JSON-структура состояния этапов планирования.

**Источники:** Формирование на основе действий пользователя состояние данных системы (выполнение действий на соответствующей странице, переход на следующий этап по кнопке).

  - Удаление cтратегических ориентиров через кнопку "Удалить"
