# Этот файл подробно описывает процесс формирование анализа 

## Формирование анализа

**Где:** Бэкенд  

**Триггер:** Событие "Формирование контекста" 

**Результат:** Сформированный аналитический набор данных для страницы "Анализ"

---

## Общий pipeline формирования анализа

1. Получение данных из БД:
   - Данные планирования (`Название мультипроекта`, `Горизонт планирования`)
   - Реестр проектов (`Название проекта`, `Статус`, `Дата начала`, `Дата окончания`, `Трудоемкость: аналитики`, `Трудоемкость: разработчики`, `Трудоемкость: тестировщики`, `Ограничения прокта`, `Текущие отклонения`, `Описание проекта`)
   - Зависимости проектов (`Основной проект`, `Зависимый проект`)
   - Портфельные ограничения (`Лимит: аналитики`, `Лимит: разработчики`, `Лимит: тестировщики`, `Критический срок`)

2. Валидация и нормализация данных

3. Последовательное формирование аналитических блоков:
   1. Граф зависимостей проектов
   2. Список выявленных рисков
   3. Анализ реализуемости портфеля по срокам
   4. Анализ загрузки и дефицита ресурсов по ролям
   5. Анализ текущих отклонений
   6. ИИ-объяснения
   7. ИИ-рекомендации

4. Сохранение результатов

---

### 1. Граф зависимостей проектов

**Определение:** Граф, отображающий проекты как узлы и зависимости как направленные ребра. Атрибуты узлов: `Статус проекта` (Завершено, В работе, Не начато).

**Входные данные:**
(поступают из БД, ранее были введены пользователем через интерфейс)
- `Реестр проектов`
- `Зависимости проектов`

Формат:
```json
projects: [
    { 
        "name": name, 
        "status": status
    }
]
dependencies: [
    { 
        "mainProject": mainProject, 
        "dependentProject": dependentProject 
    }
]
```

**Алгоритм:**
1. Создание узлов 
```json
nodes = [
  {
    "id": project.id, 
    "name": project.name,
    "status": project.status
  }
]
```
2. Создание ребер
```json
edges = [
  {
    "source": dep.mainProject,
    "target": dep.dependentProject
  }
]
```
3. Определение статуса узлов
    - Завершено → зеленый
    - В работе → синий
    - Не начато → серый

**Формат результата:**
```json
{
  "nodes": [
    { 
        "id": "id123", 
        "name": "P1", 
        "status": "В работе"
    }
  ],
  "edges": [
    { 
        "source": "P1", 
        "target": "P2" 
    }
  ]
}
```

**Особенности:**
- запрещены циклы
- обязательная проверка существования проектов

### 2. Список выявленных рисков

**Определение:** Список рисков, выявленных на основе анализа данных

**Входные данные:**
- Реестр проектов (`Название проекта`, `Статус`, `Дата начала`, `Дата окончания`, `Трудоемкость: аналитики`, `Трудоемкость: разработчики`, `Трудоемкость: тестировщики`, `Ограничения прокта`, `Текущие отклонения`, `Описание проекта`)
- Зависимости проектов (`Основной проект`, `Зависимый проект`)
- Портфельные ограничения (`Лимит: аналитики`, `Лимит: разработчики`, `Лимит: тестировщики`, `Критический срок`)
- Данные планирования (`Название мультипроекта`, `Горизонт планирования`)

---

**Используемые поля:**
```json
{
  "planningData": {
    "portfolioName": "Развитие ИС компании",
    "planningHorizon": "31.07.2026"
  },
  "projects": [
    {
      "id": "P1",
      "name": "Модуль авторизации",
      "status": "В работе",
      "startDate": "01.03.2026",
      "endDate": "30.04.2026",
      "workload": {
        "analysts": 120,
        "developers": 300,
        "testers": 80
      },
      "constraints": "Нельзя переносить релиз",
      "deviations": "Отставание на 10 дней",
      "description": "Разработка модуля авторизации"
    }
  ],
  "dependencies": [
    {
      "mainProject": "P2",
      "dependentProject": "P1",
      "description": "Проект P2 зависит от проекта P1"
    }
  ],
  "portfolioConstraints": {
    "analystsLimit": 500,
    "developersLimit": 900,
    "testersLimit": 300,
    "criticalDeadline": "31.07.2026"
  }
}
```
---

**Валидация данных:**
1. В `Реестре проектов` есть хотя бы один проект.
2. У каждого проекта заполнены:
    - название;
    - статус;
    - дата начала;
    - дата окончания;
    - трудоемкость по ролям.
3. Все даты приводятся к единому формату.
4. Дата начала проекта не позже даты окончания.
5. Все зависимости ссылаются на существующие проекты.
6. В зависимостях нет циклов.
7. Лимиты ресурсов не являются отрицательными.
8. Горизонт планирования и критический срок заполнены.

Если данные не проходят проверку, риск не формируется, а процесс возвращает ошибку валидации.

#### Формирование аналитических проблем

**Проблема** — это внутренняя аналитическая доменная модель, используемая backend-слоем для промежуточного представления выявленного нарушения. Она не является публичной сущностью интерфейса и не возвращается на фронтенд напрямую.

Пример: 
```json
{
  "problemId": "problem-001",
  "type": "deadline_violation",
  "source": "timeline_analysis",
  "mainProject": "P3",
  "dependentProject": ["P1", "P2"],
  "reason": "Дата окончания проекта выходит за горизонт планирования",
  "facts": {
    "projectEndDate": "20.08.2026",
    "planningHorizon": "31.07.2026"
  }
}
```

---

##### 1) Выявление проблем по зависимостям

Backend анализирует список `Зависимости проектов`.

Для каждой зависимости:

```json
{
  "mainProject": "P2",
  "dependentProject": "P1"
}
```

проверяется:
- существует ли основной проект;
- существует ли зависимый проект;
- есть ли у основного проекта отклонения;
- завершен ли основной проект;
- не начинается ли зависимый проект раньше завершения основного.

**Правила формирования проблем:**
1. Зависимый проект зависит от проекта с отклонением
```json
{
  "type": "dependency_deviation",
  "source": "dependency_analysis",
  "mainProject": "P2",
  "dependentProject": ["P1"],
  "reason": "Зависимый проект связан с проектом, у которого есть текущее отклонение"
}
```

2. Зависимый проект начинается до завершения основного (dependentProject.startDate < mainProject.endDate)
```json
{
  "type": "dependency_date_conflict",
  "source": "dependency_analysis",
  "mainProject": "P2",
  "dependentProject": ["P1"],
  "reason": "Зависимый проект начинается раньше завершения основного проекта"
}
```

---

##### 2) Выявление проблем по срокам

Backend сравнивает даты проектов с:
- `Горизонт планирования`
- `Критический срок`

**Правила формирования проблем:**
1. Проект выходит за горизонт планирования (mainProject.endDate > planningHorizon)
```json
{
  "type": "planning_horizon_violation",
  "source": "timeline_analysis",
  "mainProject": "P3",
  "reason": "Дата окончания проекта выходит за горизонт планирования",
  "facts": {
    "projectEndDate": "20.08.2026",
    "planningHorizon": "31.07.2026"
  }
}
```

2. Проект выходит за критический срок (mainProject.endDate > criticalDeadline)
```json
{
  "type": "critical_deadline_violation",
  "source": "timeline_analysis",
  "mainProject": "P3",
  "reason": "Дата окончания проекта выходит за критический срок портфеля",
  "facts": {
    "projectEndDate": "20.08.2026",
    "criticalDeadline": "31.07.2026"
  }
}
```

3. Узкое место по срокам

Если проект:
- находится в длинной цепочке зависимостей;
- имеет отклонение;
- влияет на несколько зависимых проектов;

```json
{
  "type": "timeline_bottleneck",
  "source": "timeline_analysis",
  "mainProject": "P2",
  "dependentProject": ["P3", "P4"],
  "reason": "Проект является узким местом по срокам"
}
```

---

##### 3) Выявление проблем по ресурсам

Backend суммирует трудоемкость проектов по ролям:
`Суммарная потребность = сумма трудоемкости всех проектов по роли`

Для каждой роли рассчитывается Дефицит(Профицит): `лимит - суммарная потребность`. Например: 
```json
{
  "role": "Разработчики",
  "required": 1200,
  "limit": 900,
  "delta": -300
}
```
**Правила формирования проблем:**
1. Превышение лимита ресурсов (required > limit)
```json
{
  "type": "resource_deficit",
  "source": "resource_analysis",
  "role": "Разработчики",
  "reason": "Суммарная потребность превышает доступный лимит ресурсов",
  "facts": {
    "required": 1200,
    "limit": 900,
    "deficit": 300
  }
}
```

---

##### 4) Выявление проблем по текущим отклонениям

Backend просматривает поле `Текущие отклонения` у каждого проекта.

Если поле не пустое, происходит поиск зависимых проектов и формируется проблема:
```json
{
  "type": "project_deviation",
  "source": "deviation_analysis",
  "mainProject": "P1",
  "reason": "У проекта существует текущее отклонение",
  "facts": {
    "deviation": "Отставание на 10 дней"
  },
  "propagation": {
    "hasPropagation": true,
    "dependentProject": ["P3", "P4"]
  }
}
```

#### Определение уровня влияния риска
После формирования проблемы backend определяет уровень влияния будущего риска.

Используются три уровня:
1. Низкий
2. Средний
3. Высокий

**Правила определения влияния**

**1. Низкий уровень**

Назначается, если:
- проблема относится к одному проекту;
- нет зависимых проектов;
- нет превышения критического срока;
- нет дефицита ресурсов.

**2. Средний уровень**

Назначается, если:
- проблема влияет на один зависимый проект;
- есть отклонение у проекта в работе;
- есть локальное превышение срока;
- есть умеренный дефицит ресурсов (до 20%). 

**3. Высокий уровень**

Назначается, если:
- проект выходит за критический срок;
- отклонение распространяется на несколько зависимых проектов;
- дефицит ресурсов значительный (> 20%);
- проблема ставит под угрозу реализуемость портфеля.

Пример результата:
```json
{
  "problemId": "problem-001",
  "type": "dependency_deviation",
  "impact": "Высокий",
  "impactReason": "Отклонение проекта P1 распространяется на зависимые проекты P2 и P3"
}
```

---

#### Преобразование проблемы в черновик риска
backend преобразует аналитическую проблему в черновой объект риска.

Черновик риска уже соответствует доменному понятию Риск, но его текстовая формулировка должна быть уточнена ИИ.

```json
{
  "riskId": "risk-001",
  "risk": "",
  "impact": "Высокий",
\  "sourceProblem": {
    "type": "dependency_deviation",
    "mainProject": "P2",
    "dependentProject": ["P1", "P3"],
    "reason": "Зависимый проект связан с проектом, у которого есть текущее отклонение", 
    "facts": {
      "deviation": "Отставание на 10 дней", 
      "constraints": "Нельзя переносить релиз"
    }
  }
}
```

#### Уточнение формулировки риска с помощью ИИ
ИИ используется только после того, как backend уже определил:
- тип проблемы;
- источник проблемы;
- проект или роль;
- связанные проекты;
- факты;
- уровень влияния.

ИИ не должен изменять расчетные факты и уровень влияния.

Вход для ИИ:
```json
{
  "riskId": "risk-001",
  "risk": "",
  "impact": "Высокий",
  "problem": {
    "type": "dependency_deviation",
    "mainProject": "P2",
    "dependentProject": ["P1", "P3"],
    "reason": "Зависимый проект связан с проектом, у которого есть текущее отклонение", 
    "facts": {
      "deviation": "Отставание на 10 дней", 
      "constraints": "Нельзя переносить релиз"
    }
  }
}
```

**Промпт:** 

```
Ты формируешь текст риска для системы стратегического планирования мультипроекта.

Используй только переданные данные.
Не добавляй новые проекты, роли, сроки, ограничения или причины.
Не изменяй уровень влияния риска.
Формулировка должна:
- быть краткой;
- начинаться со слова "Риск";
- описывать возможное последствие и его причину;
- быть пригодной для отображения на фронтенде.

Входные данные:

```json
{
  "riskId": "risk-001",
  "risk": "",
  "impact": "Высокий",
  "problem": {
    "type": "dependency_deviation",
    "mainProject": "P2",
    "dependentProject": ["P1", "P3"],
    "reason": "Зависимый проект связан с проектом, у которого есть текущее отклонение",
    "facts": {
      "deviation": "Отставание на 10 дней",
      "constraints": "Нельзя переносить релиз"
    }
  }
}

Верни только JSON:

{
  "risk": "..."
}
```

---

#### Валидация ответа ИИ
Backend проверяет ответ ИИ перед сохранением.

Проверки:
1. Ответ является валидным JSON.
2. Поле risk не пустое.
3. В тексте риска нет проектов, которых нет в исходных данных.
4. В тексте риска нет ролей, которых нет в исходных данных.
5. ИИ не добавил новых фактов.

Если ответ ИИ не прошел проверку:
- возвращается ошибка "Некорректный ответ ИИ".

#### Формирование готового объекта риска
После валидации backend формирует итоговый объект риска.

Итоговый формат соответствует термину Риск:

```json
{
  "id": "risk-001",
  "risk": "Риск задержки проекта P2 из-за отклонения проекта P1",
  "impact": "Высокий",
}
```

Для внутреннего хранения дополнительно сохраняются технические поля:

```json
{
  "riskId": "risk-001",
  "risk": "Риск задержки проекта P2 из-за отклонения проекта P1",
  "impact": "Высокий",
  "problem": {
    "type": "dependency_deviation",
    "mainProject": "P2",
    "dependentProject": ["P1", "P3"],
    "reason": "Зависимый проект связан с проектом, у которого есть текущее отклонение",
    "facts": {
      "deviation": "Отставание на 10 дней",
      "constraints": "Нельзя переносить релиз"
    }
  }
}
```

#### Возврат списка рисков
После обработки всех проблем backend возвращает список рисков в составе результата анализа:
```json
{
  "risks": [
    {
      "id": "risk-001",
      "risk": "Риск задержки проекта P2 из-за отклонения проекта P1",
      "impact": "Высокий",
    },
    {
      "id": "risk-002",
      "risk": "Риск дефицита ресурсов по роли Разработчики",
      "impact": "Средний",
    }
  ]
}
```

На фронтенде эти данные отображаются в блоке `Выявленные риски`.

проекты
зависимости

4. Анализ загрузки и дефицита ресурсов
Определение

Сравнение потребности ресурсов с лимитами

Входные данные
проекты (трудоемкость)
лимиты
Логика формирования
Алгоритм
Суммирование:
total_dev = sum(project.dev for project in projects)
Сравнение:
deficit = total_dev - limit_dev
Определение пиков:
группировка по времени (по месяцам)
Формат результата
{
  "resources": [
    {
      "role": "Разработчики",
      "required": 1200,
      "limit": 1000,
      "delta": -200
    }
  ]
}
Особенности
отрицательное значение → дефицит
положительное → профицит
